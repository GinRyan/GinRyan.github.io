#面向切面编程

定义横切关注点，可以在主要业务以外增加横切面逻辑代码，使用公共功能，并且不需要修改原来的业务代码，实现横切关注点的模块化与主要业务的解耦。

好处：

  • 每个事务的逻辑位于同一个位置，代码不分散，便于维护升级。
  • 业务模块更加简洁，与横切面逻辑完全解耦，只包含业务代码。 AOP中的概念：
  • 切面(Aspect)：横切关注点,是被模块化的特殊对象
  • 通知(Advice)：切面必须要完成的工作
  • 目标(Target)：被通知的对象
  • 代理(Proxy)：向目标对象应用通知之后创建的代理对象，此时已组合了切面对象和目标对象的新对象。
  • 连接点(JoinPoint)：程序执行的某个特定位置。例如：Object#toString()
    方法执行前的连接点执行点就是Object#toString()，方位为执行前的位置
  • 切点(PointCut)：每个类都有多个连接点。实际上一个类所有的方法都是连接点，也就是说，连接点就是程序类当中客观存在的事务。AOP通过切点定位到特定的连接点切点和连接点不是一对一的关系，一个切点可以匹配多个连接点。通过PointCut接口描述。

从上面的这个用于切面的类我们可以看到几个不一样的地方，首先这个类除了已经有@Component加入IoC容器以外，还有一个@Aspect注解，将这个类标记为切面对象。这个切面中的方法中有一个@Before注解，注解参数中有切面表达式。
所以我们看，@Aspect可以将某一个类定义为切面对象，并且必须在这个切面对象当中的方法加上切面表达式来指定这个切面关注了哪个连接点。@Before就是前置环绕注解，其参数的切面表达式的格式也很简单，主要就是为了指定在执行时关注哪个连接点。格式为：
“execution ( [方法的全局签名] )”，在上例中，被关注的连接点的方法签名就是

public int org.ginryan.aop.ArithmeticCalculator.add(int , int )

那么这个前置环绕就关注了这个add方法。当从IoC容器中获得这个Bean的实例时，就已经加入了这个切面。如果我需要将这个类当中所有符合这个方法签名的方法都关注，那么我们可以改用通配符来写方法签名，那么整个切面表达式就是：

execution (public int org.ginryan.aop.ArithmeticCalculator.*(int , int ))

原本方法名是add，现在可以改为通配符来匹配所有符合这个签名的方法。

同理，这个原则也可以适用在类名、返回类型和访问标识符上。
例：

execution (* org.ginryan.aop.*.*(int , int ))
